# 쿠키
### 쿠키의 개념

      웹 브라우저가 웹 서버로부터 정보를 받아 메모리 혹은 디스크에 저장하는 키와 값이 들어있는 작은 데이터 파일이다.
    
### 이름의 유래는?
    
      cookie는 유닉스에서 쓰이는 매직 쿠키에서 유래됐다.
    
      매직 쿠키는 바뀌지 않은 채 주고 받는 데이터 패킷을 의미한다.
    
      매직 쿠키는 포춘 쿠키에서 유래됐다.
    
      포춘 쿠키는 안에 메세지가 들어 있는 실제 먹는 쿠키이다.
        
      ---
    
      [https://en.wikipedia.org/wiki/HTTP_cookie](https://en.wikipedia.org/wiki/HTTP_cookie)
    
### 쿠키의 동작 방식
    1. 클라이언트는 페이지를 요청합니다.
    2. 서버는 쿠키를 생성한 후 http 헤더에 포함시켜 전달합니다. (브라우저에서 생성할 수도 있음)
        - HTTP 헤더에 어떻게 포함되어 있을까?

          쿠키이름=쿠키값; Domain=도메인값; Path=경로값; Expires=GMT형식의 만료일시

    3. 브라우저가 종료되어도 쿠키 만료 기간이 있다면 쿠키는 사라지지 않고 브라우저에 저장되어 있습니다.
        - 브라우저에 어디에 저장한다는 것일까?

          세션 쿠키는 메모리에 저장되어 있다가 브라우저가 종료되면 사라진다.

          persistent cookie의 경우 크롬은 SQLite 파일에 저장한다.

          그리고 이 SQLite 파일은

          C:\Users\jiwoo\AppData\Local\Google\Chrome\User Data\Default\Network

          경로에 저장되어 있다.

          즉 브라우저에 저장되는게 아니라 브라우저에 의해서 로컬 컴퓨터에 저장되는 것이다.
            
          ---

          데이빗 고울리 외 4명, HTTP 완벽 가이드, 이응준 외 1명, 도서출판인사이트, 초판 6쇄, 307p,2021

          [https://en.wikipedia.org/wiki/HTTP_cookie](https://en.wikipedia.org/wiki/HTTP_cookie)

    4. 이 후 쿠키는 클라이언트가 요청을 보낼 때 작용가능한 도메인 범위 내 모든 요청의 http 헤더에 함께 보내집니다.
    5. 서버에서 쿠키를 읽어 이전 상태 정보를 변경 할 필요가 있을 대 쿠키를 업데이트 하여 변경된 쿠키를 HTTP 헤더에 포함시켜 응답합니다.
### 왜 이러한 개념이 나왔을까?
    
      MCI의 이커머스 사이트로부터 서버가 부분적인 트랜잭션 상태를 보관하지 않고
      장바구니 기능을 제공하고 싶었다.
    
      따라서 이러한 상태를 고객의 컴퓨터에 저장하는 방안을 만들도록 넷스케이프에 요청했다.

### 왜 사용할까?
    
      기본적으로 HTTP 프로토콜 환경은 [stateless](https://www.notion.so/ccbd5c06945649328832f3719277fa95) 한 특성을 가지기 때문에 서버는 클라이언트가 누구인지 매번 확인해야 합니다.
    
      만약 로그인을 한다고 가정하면 매번 서버에게 로그인 정보를 보내야 합니다.
    
      이 과정이 개발하기 번거롭고 힘듭니다.
    
      반면 쿠키를 사용하면 한 번 만들어진 쿠키는 없어지기 전까지 쿠키 매커니즘에 의해 자동으로 보내지기 때문에 개발자 입장에서 편해서 쿠키를 사용합니다.

### 어디에 쓰이는지?
    
      방문 사이트에서 로그인 시, "아이디와 비밀번호를 저장하시겠습니까?"
    
      광고 정보 트래킹 (아 이 웹브라우저를 쓰는 사람이 이런 이런 광고를 주로 보는구나)
    
      쇼핑몰의 장바구니 기능
    
      자동로그인, 팝업에서 "오늘 더 이상 이 창을 보지 않음" 체크, 쇼핑몰의 장바구니
    
### 단점은?
    
      항상 쿠키를 보내기 때문에 쿠키가 많아질 경우 네트워크에 많은 트래픽이 추가 될 수 있다. 따라서 쿠키는 최소한의 정보에만 사용해야 한다.
    
      쿠키가 탈취될 수도 있기 때문에 보안에 민감한 정보는 저장하면 안된다.(주민번호, 신용카드 번호 등등)
    
### 서버에 전송하지 않고 그냥 웹브라우저 내부에 데이터를 저장하는 방법은 없을까?
    
      localStroage나 sessionStrorage를 사용하면 된다.
    
### 왜 JSP에서 쿠키는 삭제하는 메서드를 따로 제공하지 않을까?
    
      유효 시간을 0으로 설정했을 때 브라우저가 삭제하도록 하는게 간편하기 때문이 아닐까
    
      쿠키에 삭제와 관련된 추가적인 속성을 정의할 필요가 없어서 .
    
### 쿠키가 많아지면 웹브라우저가 느려질까?
    
      쿠키도 결국 컴퓨터에 존재하는 하나의 파일에 불과하기 때문에 텍스트 파일이 많아진다고 웹브라우저가 느려지는게 아닌 것처럼, 쿠키가 많아진다고 웹 브라우저가 느려지지느 않을 것 같다.
    
      하지만 세션 쿠키의 경우 메모리에 저장되어 있다가 브라우저가 종료되면 사라지기 때문에 브라우저를 오래 켜논 상태로 세션 쿠키가 쌓이다 보면 웹브라우저가 느려질 것 같다.
    
### 쿠키 도메인이 .domain.com 이면 [sub.domain.com](http://sub.domain.com) [some.domain.com](http://some.domain.com) 둘 다 쿠키가 전송되는데 이 때 path는 그럼 어떻게 적용될까?
    일치하는 path에서만 쿠키가 보내진다.


# 세션
### 세션의 개념
    
    사용자 정보 파일을 브라우저에 저장하는 쿠키와 달리 세션은 서버에서 관리합니다.
    
### 세션의 동작 방식
    1. 클라이언트가 서버에 접속 시 갖고 있는 세션 ID가 없다면 세션을 생성하고 세션 ID를 발급 받음
    2. 클라이언트는 세션 ID에 대해 쿠키를 사용해서 저장하고 가지고 있음
    3. 클라리언트는 서버에 요청할 때, 이 쿠키의 세션 ID를 같이 서버에 전달해서 요청
    4. 서버는 세션 ID를 전달 받아서 세션 ID로 세션에 있는 클라이언트 정보를 가져와서 사용
    5. 클라이언트 정보를 가지고 서버 요청을 처리하여 클라이언트에게 응답
### 어디에 쓰이는지?
    
    로그인 같이 보안상 중요한 작업을 수행할 때 사용
    
### 단점은?
    
    사용자가 많아질수록 서버 메모리를 많이 차지하게 됩니다. 동접자 수가 많은 웹사이트인 경우 서버에 과부하를 주게 되므로 성능 저하의 요인이 됩니다.
    
### 세션은 웹 브라우저가 꺼지면 세션을 삭제시킨다는데 어떻게 알고 삭제시킬까?
    
    웹 브라우저가 꺼질 때 삭제되는 것은 세션이 아닌 세션ID를 저장하고 있는 쿠키다.
    
    세션ID는 세션 쿠키에 저장되기 때문에 브라우저가 종료될 때 삭제된다.
    
    서버는 세션ID를 통해서 해당하는 세션이 존재하는지 확인한다. 만약 해당하는 세션ID에 맞는 세션이 없다면 세션을 생성하고 있다면 기존 세션을 사용한다.
    
    따라서 브라우저가 종료되면 저장하고 있던 세션ID가 사라지므로 기존에 사용하던 세션에 접근할 수 없어 로그인이 풀리게 된다.
    
    단순히 접근할 수 없다 뿐이지 여전히 이전 세션ID에 대한 세션은 남아있는 셈이다.
    
    하지만 기본적으로 세션은 lastAccessedTime이라는 것이 있어서 마지막으로 접근 후 일정 시간 이내에 다시 세션에 접근하지 않는 경우 자동으로 세션을 종료하는 기능을 갖고 있다.
    
    이 시간은 직접 설정할 수 있는데 만약 0이나 그 이하로 설정하게 되면 세션이 사라지지 않아 메모리 릭이 발생할 수 있다.

### 만약 세션ID를 중간에 누가 가로챈다면 어떻게 될까?
    
    서버는 기본적으로 위와 같은 상황을 방어하기 위해 다양한 확인 장치를 한다.
    
    기본적으로는 세션ID를 클라이언트에 IP와 묶어서 같은 IP로부터온 세션ID가 아니라면 허용이 안되게끔 하기도 한다.

### 세션과 세션 레이어, HTTPS의 세션 키 등 서로 무슨 관계일까?
    
    세션은 TCP/IP 연결이 수립되고 종료될 때까지의 연결 상태를 말한다.
    OSI 7계층에서 세션 레이어란 소켓 API를 통해서 프로세스간의 세션을 열 수 있게 해주는 층이다.
    
    HTTPS에서 데이터를 암호화할 때 사용하는 세션 키는 하나의 통신 세션 동안 사용되는 키이다.
    
    세션은 일반적으로 stateful하다고 한다. 세션 기간 동안은 커뮤니케이션을 위해 둘 중 하나가 상태를 유지하기 때문이다.
    
    하지만 stateless하고 connectionless한 특징을 가진 HTTP는 이러한 세션이 굉장히 짧은 시간 유지된다. 이런 HTTP에서는 상태 유지가 필요한 기능을 구현하기 위해서 세션이라는 기법을 사용한다.
    
    여기서 말하는 세션은 서버에서 관리하는 연결 대상에 대한 데이터를 저장하는 자료구조라고 할 수 있다.
    
    결국 세션이라는 용어는 의미는 다르지만 다 비슷한 맥락에서 사용되는 것 같다.

### stateless한 이유가 서버 확장성을 위한 것인데 세션을 서버에 저장하면 결국 서버 확장성이 떨이지는 문제가 똑같이 일어나지 않을까?
